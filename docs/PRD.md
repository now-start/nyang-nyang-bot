## 1. 제품 개요

Nyang-Nyang Bot은 CHZZK 채널 커뮤니티의 참여도를 `호감도`라는 잔액형 포인트로 관리하는 Spring Boot 기반 채팅봇 및 웹 관리 서비스이다.

초기 제품의 중심 가치는 범용 채팅봇이 아니라 호감도 원장이다. 출석, 관리자 조정, 업보, 후원 룰렛, 이벤트 결과를 하나의 원장과 보유 목록으로 관리하고, 사용자와 관리자가 안전하게 조회/정정할 수 있게 한다.

## 2. 문제 정의

현재 호감도 운영은 Google Sheets, 관리자 수동 처리, 채팅 이벤트, 후원/업보 결과가 분산되기 쉽다. 이 구조는 다음 문제가 있다.

- 사용자가 본인 호감도와 변경 이력을 신뢰하기 어렵다.
- 관리자가 지급/차감/정정 근거를 추적하기 어렵다.
- 업보/쿠폰/리워드처럼 호감도로 환산되지 않는 항목을 잔액과 분리해 관리하기 어렵다.
- 후원 룰렛과 OBS 오버레이 결과가 원장과 불일치할 위험이 있다.
- OAuth, 토큰 로그, 관리자 권한 같은 운영 보안 리스크가 존재한다.

## 3. 목표

1. 시청자별 호감도 잔액, 순위, 히스토리를 단일 시스템에서 조회할 수 있게 한다.
2. CHZZK 로그인 사용자는 본인 데이터만 안전하게 조회하고, 관리자는 전체 데이터를 관리할 수 있게 한다.
3. 모든 호감도 변경은 원장 거래로 기록하고 삭제 대신 정정 거래로 보정한다.
4. 업보/쿠폰/리워드는 별도 보유 목록으로 관리하고, 환산 가능한 항목만 호감도에 반영한다.
5. 방송 중 출석체크 대상자에게 빠르게 보상을 적용할 수 있게 한다.
6. Google Sheets의 기존 호감도 데이터를 플랫폼 DB로 이관한다.
7. 치지직 후원 메시지 기반 룰렛을 서버 확정 결과로 처리하고 OBS 오버레이에 표시한다.
8. 운영자는 토큰, 로그, 배포, 롤백, 재송출을 안전하게 관리할 수 있어야 한다.

## 4. 비목표

- MVP 단계의 CHZZK 외 플랫폼 지원
- 결제, 포인트 환전, 금전성 리워드 처리
- 일반 사용자용 호감도 직접 수정 기능
- Google Sheets 고도화 또는 완전한 스프레드시트 편집기
- 초기 단계의 복수 채널 멀티테넌시
- 초기 단계의 음악 시스템, 게임 전적 연동, 범용 미니게임 전체 구현

## 5. 사용자와 권한

| 사용자 | 주요 권한 |
| --- | --- |
| 방문자 | 로그인 진입 |
| 인증 사용자 | 호감도 보드 조회, 본인 히스토리 조회, 본인 업보/룰렛 결과 조회 |
| 관리자 | 전체 히스토리 조회, 사용자 검색, 호감도/업보 수동 조정, 출석체크, 마이그레이션, 룰렛/오버레이 관리 |
| 시스템 | CHZZK 소켓 유지, 채팅/후원 이벤트 처리, scheduler 재처리 |

권한 기준:

- 사용자 식별자는 CHZZK `channelId`를 사용한다.
- 관리자 권한은 초기 MVP에서 `AuthorizationAccount.admin = true` 기준이다.
- 일반 사용자는 다른 사용자의 상세 히스토리와 업보 내역을 조회할 수 없다.

## 6. 핵심 사용자 흐름

### 6.1 호감도 조회

1. 사용자가 CHZZK OAuth로 로그인한다.
2. 사용자가 호감도 보드에서 랭킹과 본인 상태를 확인한다.
3. 사용자가 본인 호감도 변경 히스토리와 업보/쿠폰/리워드 보유 목록을 확인한다.
4. 관리자는 닉네임 부분 일치 검색으로 특정 사용자의 전체 히스토리를 확인한다.

### 6.2 관리자 수동 처리

1. 관리자가 대상 사용자를 검색한다.
2. 관리자가 호감도 조정, 업보 지급, 쿠폰 구매/사용, 정정 거래 중 하나를 선택한다.
3. 시스템이 적용 전/후 잔액과 공개 설명/관리자 메모를 확인한다.
4. 적용 결과는 원장 또는 보유 목록에 기록된다.

### 6.3 출석체크

1. 관리자가 출석체크 사이클을 시작하고 지급 점수 `N`을 입력한다.
2. 활성화 구간에 채팅한 사용자가 대상자로 수집된다.
3. 관리자가 적용하면 사이클 내 사용자당 1회만 호감도가 지급된다.
4. 관리자가 취소하면 지급 없이 사이클이 종료된다.

### 6.4 후원 룰렛과 OBS 오버레이

1. 관리자가 룰렛 테이블, 명령어, 1회 금액, 확률표를 설정한다.
2. 시스템이 CHZZK 후원 이벤트를 구독한다.
3. 후원 메시지에 `!룰렛` 같은 설정 명령어가 있으면 회차 수를 계산한다.
4. 서버가 후원 시점 스냅샷 기준으로 모든 결과를 먼저 확정한다.
5. 확정 결과를 호감도 원장과 업보/쿠폰 보유 목록에 반영한다.
6. OBS 오버레이는 확정 결과를 표시만 한다.
7. 놓친 오버레이 이벤트는 관리자가 재송출할 수 있으며 원장에는 재반영하지 않는다.

## 9. 디자인 요구사항

웹 UI:

- 호감도 보드는 현재 사용자, 주간 채팅 순위, 검색, 랭킹, 페이지네이션을 한 화면 흐름으로 제공한다.
- 마이페이지는 현재 호감도, 미확인 배지, 최근 변경, 보유 업보/쿠폰/리워드, 전체 히스토리를 분리해 보여준다.
- 관리자 화면은 대상 사용자, 처리 유형, 입력값, 적용 전/후 미리보기를 한 흐름에서 제공한다.
- 출석체크는 사이클 상태, 지급 점수, 수집 대상자 수, 적용/취소 액션을 명확히 보여준다.
- 룰렛 관리 화면은 확률 합계, 꽝 필수 여부, 활성화 가능 여부, 시뮬레이션 결과를 즉시 확인할 수 있어야 한다.
- 오버레이 토큰은 원문 1회 표시, 복사, 재발급, 폐기 UX를 제공한다.

OBS 오버레이:

- 이벤트가 없을 때는 완전 투명 상태를 유지한다.
- 1920x1080과 1280x720에서 결과 라벨이 읽혀야 한다.
- 단회차는 서버 확정 결과로 감속/정지하는 애니메이션을 제공한다.
- 다회차는 관리자 설정 N회까지 순차 표시하고 초과분은 요약한다.
- missed/replay는 원장 재반영 없이 화면 표시 이벤트만 처리한다.
- 토큰 오류나 네트워크 오류는 기본 방송 화면에 노출하지 않는다.

상세 와이어프레임은 [디자인 와이어프레임](spec/wireframes.md)을 따른다.

## 10. 출시 기준

MVP 출시 기준:

- P0 요구사항이 사용자/관리자 주요 플로우에서 동작한다.
- OAuth state와 민감 토큰 로그 문제가 해결된다.
- 호감도 변경은 원장과 잔액이 같은 트랜잭션으로 처리된다.
- 관리자 정정 거래가 삭제 없이 동작한다.
- 출석체크 사이클이 중복 지급 없이 동작한다.
- Google Sheets 마이그레이션 검증 절차가 있다.
- 호감도 보드와 관리자 모달이 [디자인 와이어프레임](spec/wireframes.md)의 MVP 구조를 충족한다.
- `./gradlew test`가 통과한다.

P1 출시 기준:

- CHZZK 후원 이벤트 기반 룰렛이 idempotency를 보장한다.
- 룰렛 결과가 서버에서 먼저 확정되고 원장/보유 목록에 반영된다.
- OBS 오버레이 토큰이 안전하게 발급/재발급/폐기된다.
- missed/replay가 원장 재반영 없이 동작한다.
- OBS 오버레이 idle/displaying/summary 상태가 [OBS 오버레이 디자인 명세](spec/overlay-design.md)를 충족한다.
