## 1. 범위

이 문서는 Nyang-Nyang Bot의 웹 화면, 사용자/관리자 UI, 공통 컴포넌트, 상태 표현, 반응형 기준을 정의한다. OBS 브라우저 소스 화면은 [OBS 오버레이 디자인 명세](overlay-design.md)를 따른다.

현재 구현은 `templates/index.html`, `static/css/favorite-list.css`, `static/js/favorite-list.js`, `templates/fragments/*`를 중심으로 호감도 보드, 주간 채팅 순위, 관리자 호감도 조정 모달, 출석체크 모달을 제공한다. 신규 화면은 현행 사용성을 유지하되 클린 아키텍처 adapter 경계에 맞춰 request/response DTO와 화면 상태를 분리한다.

## 2. 디자인 원칙

- 화면은 방송 커뮤니티 운영 도구로서 빠른 조회와 반복 작업을 우선한다.
- 관리자 기능은 장식보다 밀도, 검증, 적용 전 미리보기를 우선한다.
- 일반 사용자는 본인 상태, 최근 변화, 보유 항목을 빠르게 이해할 수 있어야 한다.
- 음수 호감도, 정정 거래, 미확인 항목은 숨기지 않고 명확히 표시한다.
- 금전성, 환전성, 현금 등가 표현은 사용하지 않는다.
- 모든 상태 변경 UI는 성공/실패 피드백을 즉시 제공한다.
- 버튼/입력/테이블/모달은 작은 화면에서도 텍스트가 겹치지 않아야 한다.

### 2.1 디자인 시스템 규칙

이 프로젝트의 웹 UI는 `방송 커뮤니티 운영 도구`로 보이게 만든다. 화면은 귀엽거나 이벤트성인 느낌보다, 운영자가 반복적으로 조회하고 처리하기 쉬운 조용한 대시보드 톤을 우선한다.

#### 2.1.1 시각 언어

- 기본 테마는 어두운 운영 화면으로 유지한다.
- 주 색상은 청록 계열 1개, 관리자/주의 액션은 주황 계열 1개만 사용한다.
- 위험/실패/음수 상태는 붉은 계열을 사용하되, 일반 강조색으로 쓰지 않는다.
- 배경은 전체 화면에 한 번만 적용한다. 개별 섹션마다 별도 그라데이션을 반복하지 않는다.
- 장식용 blob, orb, bokeh, 과한 glow는 사용하지 않는다.
- 카드형 표면은 정보 묶음, 모달, 반복 목록 행에만 사용한다. 페이지 섹션 전체를 카드 안에 다시 넣지 않는다.
- border radius는 기본 `12px`, 큰 패널/모달은 `16px`, pill badge만 `999px`를 사용한다.

#### 2.1.2 타이포그래피

- 기본 폰트는 웹 UI 전체에서 하나의 stack을 사용한다.
  - 권장: `"Archivo", "Space Grotesk", "Apple SD Gothic Neo", "Malgun Gothic", sans-serif`
- OBS 오버레이도 같은 계열의 fallback을 사용하되, 방송 가독성을 위해 굵기와 크기만 키운다.
- 본문은 `14px` 또는 `15px`, 보조 텍스트는 `12px` 또는 `13px`, 화면 제목은 `28px`에서 `36px` 사이로 제한한다.
- 모달/테이블/툴바 내부에는 hero 크기 텍스트를 쓰지 않는다.
- letter spacing은 상태 label이나 작은 badge에만 제한적으로 사용한다.
- 긴 닉네임, 설명, 결과 라벨은 ellipsis 또는 2줄 clamp 규칙을 둔다.

#### 2.1.3 레이아웃

- 기본 화면 폭은 `1080px`, 관리자 밀집 화면은 최대 `1280px`까지 허용한다.
- 메인 화면은 `상단 툴바 -> 요약/필터 -> 목록/작업 영역` 순서로 구성한다.
- hero형 큰 소개 영역은 로그인/공개 랜딩이 아닌 운영 화면에서는 사용하지 않는다.
- 목록 화면에서 검색/필터는 결과 목록보다 시각적으로 작아야 한다.
- 모든 고정 포맷 UI는 `grid-template-columns`, `minmax(0, ...)`, `min-height` 등으로 크기를 안정화한다.
- 모바일에서는 1열 흐름을 기본으로 하고, 표는 카드형 행 또는 명시적 가로 스크롤 중 하나를 선택한다.

#### 2.1.4 색상과 상태

| 용도 | 규칙 |
| --- | --- |
| Primary | 저장, 적용, 발급, 활성화 |
| Admin | 관리자 전용 실행, 주황 계열 |
| Warning | 음수 가능, 비활성화, 정책 위반 근접 상태 |
| Danger | 삭제, 폐기, 실패, 음수 잔액 강조 |
| Success | 성공 toast 또는 완료 badge에만 제한 |
| Neutral | 조회, 닫기, 취소, 필터 |

- 색상만으로 상태를 구분하지 않는다. label, badge, 보조 문구를 같이 둔다.
- 음수 호감도는 숫자 색상, badge, 보조 설명 중 최소 2개를 함께 사용한다.
- 관리자 전용 액션은 일반 액션과 색상/위치가 구분되어야 한다.

#### 2.1.5 버튼

- 버튼 종류는 `primary`, `secondary`, `ghost`, `danger`, `icon`으로 제한한다.
- 같은 화면의 버튼 높이는 같은 줄에서 동일해야 한다.
- 주요 액션은 오른쪽 끝 또는 모달 하단 오른쪽에 둔다.
- 취소/닫기 액션은 주요 액션보다 낮은 대비를 사용한다.
- 닫기, 복사, 새로고침, 재송출, 펼침은 가능하면 icon 버튼으로 제공하고 tooltip을 둔다.
- 텍스트 버튼은 명령이 명확한 경우에만 사용한다.
- loading, disabled, hover, focus-visible 상태를 모두 정의한다.

#### 2.1.6 입력과 폼

- 입력 필드는 항상 label 또는 시각적으로 대응되는 짧은 제목을 가진다.
- placeholder는 예시로만 사용하고 label을 대체하지 않는다.
- 숫자 입력은 단위와 최소/최대 조건을 화면에 드러낸다.
- 호감도 증감값은 `+10`, `-5`처럼 부호가 보이게 표시한다.
- 검증 오류는 해당 입력 근처에 표시하고, toast만으로 끝내지 않는다.
- 관리자 메모와 공개 설명은 시각적으로 구분한다. 관리자 메모는 일반 사용자 화면에 노출되지 않는다는 표시를 둔다.

#### 2.1.7 목록과 데이터 표시

- 운영 목록은 장식 카드보다 조밀한 행 레이아웃을 우선한다.
- 행 hover는 약한 border/배경 변화만 사용하고, 큰 이동이나 확대를 쓰지 않는다.
- 숫자는 tabular style 또는 고정 폭 영역에 배치한다.
- 날짜/시간, 상태, 금액/점수는 항상 같은 열 위치에 둔다.
- 펼침 상세 영역은 원 행의 하위 정보처럼 보여야 하며, 독립 카드처럼 과하게 뜨지 않는다.

#### 2.1.8 모달

- 모달 구조는 항상 `header -> context -> form/list -> preview/status -> actions` 순서를 따른다.
- 모달 제목, 대상 사용자, 현재 상태를 상단에서 바로 확인할 수 있어야 한다.
- 액션 영역은 하단 고정 또는 하단 정렬로 통일한다.
- 모달 내부에 또 다른 카드형 패널을 중첩하지 않는다. 구분이 필요하면 section border나 divider를 사용한다.
- 모달 폭은 일반 작업 `720px`, 복합 관리자 작업 `960px`까지 허용한다.
- 화면 높이를 넘는 경우 body가 아니라 모달 내부 content만 스크롤한다.

#### 2.1.9 OBS 오버레이 예외

- OBS 오버레이는 방송 가독성 때문에 글자 크기, shadow, 대비를 웹 UI보다 강하게 쓸 수 있다.
- 단, 색상 토큰과 폰트 계열은 웹 UI와 공유한다.
- 오버레이는 배경이 투명해야 하며, 웹 관리 화면의 배경 그리드나 페이지 패턴을 가져오지 않는다.
- 오버레이 패널 radius, border, accent 색상은 웹 UI와 같은 계열을 사용한다.

#### 2.1.10 금지 규칙

- 관리자 화면에 마케팅형 hero를 만들지 않는다.
- 텍스트가 들어간 rounded rectangle을 과도하게 badge처럼 쓰지 않는다.
- 같은 역할의 버튼에 서로 다른 색/크기/radius를 쓰지 않는다.
- 모달 안에 모달처럼 보이는 큰 카드 패널을 중첩하지 않는다.
- 색상 팔레트를 한 가지 계열로만 구성하지 않는다.
- 모바일에서 텍스트가 버튼/카드 밖으로 넘치는 상태를 허용하지 않는다.
- Bootstrap 기본 스타일과 커스텀 스타일이 충돌하는 상태로 두지 않는다.

#### 2.1.11 구현 체크리스트

새 UI 또는 UI 수정은 다음 조건을 만족해야 한다.

- 공통 CSS token을 먼저 사용하고, 화면별 새 색상/spacing을 임의로 추가하지 않는다.
- 동일 화면 안의 버튼 높이, radius, padding이 일관된다.
- desktop `1280px`, mobile `360px`에서 텍스트 겹침이 없다.
- hover/focus/loading/disabled 상태가 정의되어 있다.
- 관리자 전용 액션과 일반 조회 액션이 시각적으로 구분된다.
- OBS 오버레이는 `1920x1080`, `1280x720`에서 결과 텍스트가 잘리지 않는다.

## 4. 공통 레이아웃

### 4.1 데스크톱

- 기본 content width는 1080px 이상을 기준으로 하되, 관리자 테이블 화면은 1280px까지 확장 가능하다.
- 상단에는 화면 제목, 현재 사용자/권한, 주요 액션을 배치한다.
- 목록 화면은 검색/필터 영역과 결과 테이블을 분리한다.
- 관리자 화면은 같은 행에서 대상자, 적용값, 결과 미리보기를 볼 수 있어야 한다.

### 4.2 모바일

- 최소 360px 폭에서 주요 텍스트와 버튼이 겹치지 않아야 한다.
- 테이블은 카드형 행 또는 가로 스크롤 중 하나로 명확히 선택한다.
- 관리자 다중 입력 폼은 한 열로 쌓는다.
- 모달은 화면 높이를 넘으면 내부 스크롤을 사용한다.

### 4.3 현행 화면 유지 기준

현행 호감도 보드는 다음 요소를 유지한다.

- hero 영역의 현재 사용자/채널 표시
- 주간 채팅 순위 ticker
- 닉네임 검색
- 호감도 목록과 페이지네이션
- 관리자 전용 출석체크/데이터 동기화/호감도 수정 버튼
- toast, loading, modal fragment

개선 필요:

- 일반 사용자에게 관리자 전용 버튼은 숨기거나 disabled + tooltip 처리한다.
- 히스토리 펼침은 본인 또는 관리자만 가능해야 한다.
- 음수 호감도 행은 색상과 보조 텍스트로 명확히 구분한다.
- 검색 placeholder는 한국어로 통일한다.

## 5. 공통 컴포넌트

### 5.1 버튼

| 종류 | 용도 |
| --- | --- |
| Primary | 저장, 적용, 발급, 재송출 같은 주요 동작 |
| Secondary | 취소, 닫기, 보조 이동 |
| Danger | 폐기, 비활성화, 음수 차감 같은 위험 동작 |
| Ghost | 필터 초기화, 상세 보기, 보조 액션 |
| Icon | 새로고침, 복사, 재발급, 닫기 |

규칙:

- 상태 변경 버튼은 loading/disabled 상태를 제공한다.
- 위험 동작은 확인 모달 또는 명확한 확인 문구를 사용한다.
- 버튼 텍스트는 한 줄에서 잘리지 않게 최소 너비와 줄바꿈 기준을 둔다.

### 5.2 입력

- 필수값은 label과 오류 메시지로 표시한다.
- 숫자 입력은 최소/최대/정수 여부를 즉시 검증한다.
- 호감도 증감값은 `+10`, `-5`처럼 부호를 명확히 표시한다.
- 자유 입력 업보는 라벨, 공개 설명, 관리자 메모를 모두 요구한다.

### 5.3 테이블/목록

- 기본 정렬은 최신순 또는 랭킹순을 명확히 표시한다.
- 히스토리는 최대 50건 페이지 단위로 표시한다.
- 원장/업보/룰렛 결과 목록은 상태 badge를 사용한다.
- 정정 거래는 원본 거래 링크 또는 참조 표시를 제공한다.

### 5.4 모달

- 모달은 제목, 대상, 입력 영역, 미리보기, 액션 영역 순서로 구성한다.
- 적용 전/후 호감도를 항상 표시한다.
- 취소/닫기와 적용 버튼은 시각적으로 구분한다.
- 모달 닫기 시 미저장 입력이 있으면 확인이 필요하다.

### 5.5 토스트와 에러

- 성공 토스트는 짧게 표시한다.
- 실패 토스트는 사용자가 다음 행동을 알 수 있게 작성한다.
- 권한 없음, 네트워크 실패, 검증 실패는 구분한다.
- 장시간 작업은 loading overlay 또는 버튼 내 loading 상태를 표시한다.

## 6. 호감도 보드

목적:

- 인증 사용자가 호감도 랭킹과 주간 채팅 순위를 확인한다.
- 관리자가 사용자를 검색하고 빠르게 수동 조정/히스토리를 확인한다.

필수 요소:

- 현재 사용자 badge
- 주간 채팅 순위
- 닉네임 검색
- 호감도 목록
- 페이지네이션
- 본인/관리자 히스토리 펼침
- 관리자 전용 호감도 수정 버튼

상태:

| 상태 | 표시 |
| --- | --- |
| 목록 없음 | 빈 상태 문구와 검색 초기화 버튼 |
| 검색 결과 없음 | 검색어와 함께 결과 없음 표시 |
| 음수 호감도 | 빨간색 계열 score, `음수` badge 또는 경고 스타일 |
| 히스토리 로딩 | 행 내부 skeleton 또는 `불러오는 중...` |
| 히스토리 권한 없음 | toast + 상세 영역 닫힘 |

## 7. 마이페이지

목적:

- 일반 사용자가 본인 호감도, 업보/쿠폰/리워드, 룰렛 결과, 정정 거래를 확인한다.

필수 섹션:

- 현재 호감도 카드
- 미확인 배지
- 최근 호감도 변경
- 보유 업보/쿠폰/리워드 목록
- 최근 룰렛 결과
- 전체 히스토리

보유 목록:

- 기본 정렬은 `createdAt DESC`.
- 기본 필터는 `OWNED`.
- 상태 필터: `OWNED`, `USED`, `CONVERTED`, `CORRECTED`.
- 환산 불가 항목은 호감도 잔액과 분리해 표시한다.

미확인 배지:

- `lastSeenAt` 이후 신규 보유 항목과 정정 거래 수를 표시한다.
- 페이지 진입 시 읽음 처리한다.
- 정확한 알림 보장이 아니라 best-effort 인지 보조로 취급한다.

## 8. 관리자 호감도/업보 처리

목적:

- 관리자가 호감도 지급/차감, 업보 지급, 쿠폰 구매/사용, 정정 거래를 안전하게 처리한다.

필수 UI:

- 대상 사용자 검색
- 현재 호감도 표시
- 처리 유형 선택
- 템플릿 선택 또는 자유 입력
- 증감값 입력
- 공개 설명 입력
- 관리자 메모 입력
- 적용 전/후 미리보기
- 적용 버튼
- 최근 처리 이력

검증:

- 대상 사용자 필수.
- 자유 입력 시 라벨/공개 설명/관리자 메모 필수.
- 일반 사용자 화면에 관리자 메모가 노출되지 않아야 한다.
- 음수 잔액 생성은 관리자만 가능하며 적용 전 경고한다.
- 정정 거래는 원본 거래 선택 또는 참조 입력이 필요하다.

## 9. 출석체크 UI

목적:

- 관리자가 방송 중 출석체크 사이클을 시작하고 채팅 참여자를 수집해 일괄 지급한다.

필수 UI:

- 사이클 시작 버튼
- 지급 점수 입력
- 수집 대상자 목록
- 대상자 수
- 새로고침/자동 갱신 표시
- 적용 버튼
- 취소 버튼

상태:

| 상태 | 표시 |
| --- | --- |
| 대기 | 시작 버튼 활성 |
| 수집 중 | 대상자 목록과 polling 상태 표시 |
| 적용 중 | 버튼 disabled, loading |
| 적용 완료 | 지급 대상 수와 총 지급량 toast |
| 취소 | 지급 없음 toast |

## 10. 룰렛 관리 UI

목적:

- 관리자가 룰렛 명령어, 1회 금액, 항목, 확률, 결과 타입을 관리한다.

필수 UI:

- 룰렛 테이블 목록
- 활성/비활성 토글
- 명령어 입력
- 1회 금액 입력
- 최대 순차 애니메이션 회차 입력
- 항목 편집 테이블
- 확률 합계 표시
- 꽝 필수 검증 표시
- 1만 회 시뮬레이션 미리보기
- 공개 확률표 미리보기

검증:

- 확률 합계가 100%가 아니면 활성화 불가.
- `꽝`이 없거나 0%이면 활성화 불가.
- 항목이 많으면 OBS/모바일 가독성 경고 표시.
- 소수점 넷째 자리까지 입력 가능.

## 11. 명령어 관리 UI

목적:

- 관리자가 채팅 명령어를 생성, 수정, 비활성화하고 충돌과 템플릿 변수를 저장 전에 확인한다.

P1 필수 UI:

- 명령어 목록 테이블
- 유형 필터: `TEXT`, `TRIGGER`
- 활성 상태 필터
- 트리거 입력
- 메시지 템플릿 입력
- 쿨타임 입력
- 템플릿 미리보기
- 충돌 검증 결과
- 비활성화 확인 액션

P2 UI:

- `TIMER` 유형 필터
- 타이머 주기와 최소 채팅 수 입력

검증/경고:

- 같은 트리거가 있으면 활성 여부와 관계없이 저장 또는 활성화 불가.
- 허용되지 않은 템플릿 변수가 있으면 저장 버튼을 비활성화한다.
- `TIMER` UI를 제공할 때는 `timer_min_chat_count=0`을 허용하지 않는다.
- 모바일 360px에서는 트리거, 활성 토글, 충돌 상태가 겹치지 않아야 한다.

## 12. 오버레이 토큰 관리 UI

목적:

- 관리자가 OBS 오버레이 토큰을 발급, 재발급, 폐기한다.

필수 UI:

- 토큰 목록
- 이름
- 활성 여부
- 마지막 사용 시각
- 발급자
- 발급 버튼
- 재발급 버튼
- 폐기 버튼
- OBS URL 복사 버튼

보안 UX:

- 토큰 원문은 발급/재발급 직후 한 번만 표시한다.
- 토큰 영역은 자동 숨김 또는 명확한 경고를 제공한다.
- 복사 버튼은 토큰 원문이 표시되는 순간에만 활성화한다.
- 재발급/폐기는 확인 모달을 사용한다.

## 13. 접근성과 반응형

- 모든 입력에는 label이 있어야 한다.
- 모달은 ESC와 닫기 버튼을 지원한다.
- 키보드 포커스 순서는 화면 순서와 일치해야 한다.
- 색상만으로 상태를 구분하지 않고 badge/text를 함께 사용한다.
- 터치 대상은 최소 40px 높이를 권장한다.
- 애니메이션은 기능 이해를 돕는 수준으로 제한한다.

## 14. 테스트 기준

- 호감도 보드는 데스크톱과 모바일에서 텍스트가 겹치지 않는다.
- 일반 사용자는 관리자 버튼을 사용할 수 없다.
- 히스토리 권한 없음은 적절히 표시된다.
- 모달에서 필수값 오류가 inline으로 표시된다.
- 음수 호감도 행은 랭킹과 마이페이지 모두에서 구분된다.
- 오버레이 토큰 원문은 재조회 화면에 표시되지 않는다.
