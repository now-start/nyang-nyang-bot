## 2. 개요

Nyang-Nyang Bot은 CHZZK 채널 커뮤니티의 참여도를 관리하기 위한 Spring Boot 기반 채팅봇 및 웹 관리 서비스이다. 핵심 제품 가치는 `호감도`라고 부르는 잔액형 포인트 제도이며, 출석, 채팅, 관리자 조정, 업보, 후원 룰렛, 이벤트 참여 결과를 하나의 원장으로 관리한다.
https://accounts.skilljar.com/accounts/login/?t=3gufixqhei80k&d=cahl60vup5xv&next=%2Fauth%2Fendpoint%2Flogin%2Fresult%3Fnext%3D%252F%26d%3Dcahl60vup5xv
최종 목표는 빵떡이처럼 스트리머가 채팅 명령어, 방송 관리, 오버레이, 미니게임, 외부 연동, 커뮤니티 이벤트를 운영할 수 있는 채팅봇 플랫폼이 되는 것이다. 다만 초기 차별화 중심은 범용 채팅봇 기능이 아니라 호감도 포인트 제도에 둔다.

## 3. 현재 구현 범위

- CHZZK OAuth 로그인 및 세션 기반 인증
- 호감도 순위 보드, 닉네임 검색, 페이지네이션
- 개인/관리자 권한 기반 호감도 히스토리 조회
- 관리자용 호감도 조정 항목 관리 및 적용
- 실시간 채팅 참여자 기반 출석체크 수집/적용
- CHZZK 채팅 소켓 연결 및 `!호감도` 채팅 명령 응답
- 주간 채팅 순위 집계 및 화면 노출
- 일반 채팅은 호감도를 지급하지 않고 주간 채팅 랭킹에만 사용
- 기존 Google Sheets 호감도 데이터 마이그레이션
- Spring Actuator, OpenAPI, Grafana 대시보드 연동 기반 운영 관측

## 4. 목표

1. 시청자별 호감도 점수를 단일 시스템에서 조회, 검색, 정렬, 이력 추적할 수 있게 한다.
2. CHZZK 로그인 사용자는 본인 데이터만 안전하게 조회하고, 관리자는 전체 데이터를 관리할 수 있게 한다.
3. 방송 중 채팅 참여자에게 출석 보상을 빠르게 적용할 수 있게 한다.
4. 기존 Google Sheets로 관리되던 호감도 데이터를 플랫폼 DB로 마이그레이션한다.
5. 업보/쿠폰/리워드는 별도 목록으로 관리하고, 환산 가능한 항목만 호감도 포인트로 전환할 수 있게 한다.
6. 업보를 관리자 페이지에서 수동 입력하고 정정할 수 있게 한다.
7. 관리자는 대출성 운영 판단에 따라 사용 가능한 업보/리워드를 수동 사용 처리하고 호감도를 음수로 만들 수 있다.
8. 이후 치지직 후원 메시지의 `!룰렛` 명령어를 감지해 OBS 오버레이 룰렛을 실행하고 결과를 호감도 원장에 반영한다.
9. 실제 채널 운영 정책에 맞춰 무료 획득, 사용처, 업보 교환표를 관리 가능한 정책 데이터로 제공한다.
10. 우선 출시는 호감도 중심 플랫폼으로 제한하고, 빵떡이식 범용 채팅봇 기능은 후순위 로드맵으로 확장한다.

## 5. 비목표

- MVP 단계의 CHZZK 외 플랫폼 지원
- 결제, 포인트 환전, 금전성 리워드 처리
- 일반 사용자용 호감도 직접 수정 기능
- Google Sheets 연동 고도화 또는 완전한 스프레드시트 편집기
- 초기 단계의 복수 채널 멀티테넌시
- 초기 단계의 음악 시스템, 게임 전적 연동, 범용 미니게임 전체 구현

## 6. 사용자와 권한

| 사용자 | 주요 권한 |
| --- | --- |
| 방문자 | `/` 접근 시 로그인으로 이동 |
| 인증 사용자 | 호감도 보드 조회, 본인 전체 히스토리 조회, 본인 업보 내역과 호감도 반영 점수 조회 |
| 관리자 | 전체 히스토리 조회, 호감도/업보 수동 조정, 출석체크, Google Sheets 마이그레이션, CHZZK 소켓 연결 |
| 시스템 | CHZZK 소켓 연결 유지, 채팅/후원 이벤트 처리, 예약 작업 실행 |

관리자 권한은 현재 `AuthorizationEntity.admin = true`를 기준으로 한다.
초기 MVP에서는 `ADMIN` 단일 관리자 권한만 사용한다. 스트리머/매니저/조회 전용 관리자 같은 세부 권한 분리는 후순위로 둔다.
사용자 고유 식별자는 CHZZK `channelId`를 사용하고, 닉네임은 표시명으로 자동 갱신한다. 히스토리에는 거래 당시 닉네임 스냅샷을 남긴다.

## 7. 핵심 사용자 흐름

### 7.1 호감도 조회

1. 사용자가 CHZZK OAuth로 로그인한다.
2. `/favorite/list`에서 호감도 보드를 조회한다.
3. 닉네임 검색과 페이지네이션으로 사용자를 찾는다.
4. 사용자는 본인 호감도/업보/쿠폰/리워드 전체 히스토리를 조회한다.
5. 사용자는 본인 업보 내역과 업보가 호감도로 변환되어 반영된 점수를 조회한다.
6. 관리자는 전체 히스토리와 전체 업보 내역을 조회한다.

### 7.2 관리자 수동 조정

1. 관리자가 대상 사용자를 검색한다.
2. 호감도 조정 항목 또는 직접 입력값을 선택한다.
3. 적용 전/후 호감도를 확인한다.
4. 변경 내역을 원장에 기록한다.

### 7.3 업보 수동 입력

1. 관리자가 업보 템플릿을 관리한다.
2. 외부 후원사이트 룰렛후원 결과를 확인한다.
3. 대상 사용자에게 업보 증감값을 수동 적용한다.
4. 잘못 적용한 경우 반대 거래로 정정한다.

상세: [호감도/업보/쿠폰 카탈로그](reward-catalog.md)

### 7.4 치지직 후원 룰렛

1. 관리자가 오버레이 토큰을 발급하고 OBS 브라우저 소스에 `/overlay/roulette#token=...` URL을 등록한다.
2. OBS 오버레이는 평소 투명 화면으로 대기한다.
3. 시스템이 치지직 후원 이벤트를 구독한다.
4. 후원 메시지에서 관리자 설정 룰렛 명령어를 감지한다.
5. 후원 금액과 룰렛 1회 금액으로 실행 회차를 계산하고, 후원자 본인을 대상자로 확정한다.
6. 서버가 활성 룰렛 테이블 버전과 확률표 기준으로 전체 결과를 먼저 확정한다.
7. 서버 확정 즉시 호감도 원장과 업보/쿠폰 히스토리에 `UPBO_ROULETTE`으로 기록한다.
8. OBS 오버레이가 HTTP 기반 이벤트 조회로 서버 확정 결과를 받아 순차 애니메이션 또는 요약으로 표시한다.

상세: [치지직 후원 룰렛과 OBS 오버레이](roulette-overlay.md)

| FR-010 | Google Sheets 포인트 마이그레이션 | P0 | 전환 기간 한정 |
| FR-011 | 업보 템플릿/자유 입력 관리 | P0 | 관리자는 템플릿을 선택하거나 라벨/증감값/메모를 자유 입력해 업보를 처리할 수 있어야 한다. |
| FR-012 | 업보 수동 적용/정정 | P0 | `UPBO_MANUAL` 원장 기록 |
| FR-013 | 관리자 수동 구매/사용 처리 | P0 | 관리자는 포인트 차감, 쿠폰 구매 처리, 즉시 사용 처리, 대출성 음수 처리를 자유롭게 실행할 수 있어야 한다. |
| FR-014 | 호감도 원장 확장 | P0 | 증감값, 잔액, 출처, 실행 주체 |
| FR-015 | 호감도 정책 관리 | P0 | 무료 획득, 사용처, 업보 교환표를 관리자가 변경할 수 있어야 한다. |
| FR-016 | 치지직 후원 이벤트 구독 | P1 | 후원자, 금액, 메시지 저장 |
| FR-017 | 후원 명령어 기반 룰렛 실행 | P1 | 관리자 설정 명령어와 1회 금액 기준으로 회차를 자동 계산하고 차단 사용자 정책과 룰렛 설정 유효성을 검증 |
| FR-018 | OBS 룰렛 오버레이 | P1 | 관리자가 발급한 오버레이 토큰으로 HTTP 페이지를 호출하고, 서버 확정 결과를 120초 유효시간 안에 관리자 설정 N회까지 순차 애니메이션, 초과 회차는 요약 표시 |
| FR-019 | 룰렛 결과 반영/조회 | P1 | 룰렛 결과는 채팅에 자동 공지하지 않고 `UPBO_ROULETTE` 원장과 사용자 히스토리로 기록 |
| FR-020 | 룰렛 확률표 관리 | P1 | 관리자는 룰렛 금액, 항목, 확률, 결과 타입, 공개 여부, 활성 여부, 꽝 필수 정책을 변경할 수 있어야 한다. |
| FR-021 | 룰렛 결과 정정 | P1 | 룰렛 결과는 삭제/취소하지 않고, 관리자는 별도 보정 거래로 정정할 수 있어야 한다. |
| FR-022 | 명령어 템플릿 관리 | P1 | 권한, 쿨타임, 변수 |
| FR-023 | 공개 채널 페이지 | P2 | 랭킹, 명령어, 이벤트 규칙 |
| FR-024 | 오버레이 토큰 관리 | P1 | 관리자는 OBS 오버레이용 토큰을 발급, 재발급, 확인, 폐기할 수 있어야 한다. |
| FR-025 | 룰렛 오버레이 재송출 | P1 | 관리자는 놓친 룰렛 이벤트를 오버레이에 재송출할 수 있어야 하며, 재송출은 원장/보유 목록을 다시 반영하지 않는다. |

## 9. 로드맵

### Phase 1. MVP 안정화

- OAuth state 검증
- 민감 토큰 로그 제거
- Google Sheets 마이그레이션 검증
- 호감도 수동 조정 개선
- 업보 템플릿 관리 및 수동 적용 관리자 페이지
- 업보 정정 거래 모델

### Phase 2. 호감도 플랫폼화

- 호감도 원장 모델 확장
- 호감도 정책 관리
- 관리자 권한 관리
- 출석체크 영속화 개선
- Google Sheets 마이그레이션 기능 제거 및 플랫폼 원장 단일화

### Phase 3. 후원 룰렛/오버레이

- 치지직 후원 이벤트 구독
- `!룰렛` 명령어 파싱
- 관리자 룰렛 확률표 관리
- 룰렛 실행 요청과 중복 방지 키 저장
- OBS 오버레이 토큰 발급/재발급/폐기
- OBS 브라우저 소스용 룰렛 오버레이
- 놓친 룰렛 이벤트 오버레이 재송출
- 룰렛 결과를 `UPBO_ROULETTE` 원장 거래로 반영

### Phase 4. 채팅봇 플랫폼 확장

- 채널별 공개 명령어 페이지
- 명령어 템플릿 변수 시스템
- 호감도 기반 미니게임 또는 이벤트 응모
- 스트리머/매니저/조회 전용 관리자 권한 체계 확장
- 장기적으로 다중 채널 운영 모델 검토
