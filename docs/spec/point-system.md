## 1. 핵심 정의

`호감도`는 치즈냥 채널 커뮤니티의 참여도를 표현하는 잔액형 포인트이다. 출석, 관리자 보상, 업보, 후원 룰렛, 이벤트 참여 결과는 모두 호감도 원장 또는 업보/쿠폰/리워드 보유 목록으로 기록한다.

기본 원칙:

- 사용자의 고유 식별자는 CHZZK `channelId`를 사용한다.
- 닉네임은 표시명이며, 로그인/채팅/후원 이벤트에서 최신 닉네임이 확인되면 갱신한다.
- 히스토리와 원장에는 거래 당시 닉네임 스냅샷을 저장한다.
- 호감도는 숫자 포인트이고, 업보/쿠폰/리워드는 별도 보유 목록으로 관리한다.
- 호감도로 환산 가능한 업보만 교환값을 가진다.
- 호감도로 환산되지 않는 업보는 호감도 잔액에 포함하지 않고 업보 목록에만 표시한다.
- `호감도 +N`처럼 결과가 명확한 포인트 쿠폰은 획득 즉시 자동 전환해 호감도 잔액에 반영한다.
- 시참권, 쿠폰, 방송 리워드처럼 사용처가 있는 항목은 자동 전환하지 않고 업보/쿠폰 목록에 보관한다.
- 호감도 잔액은 음수를 허용하지만, 음수 잔액을 만드는 거래는 관리자만 실행할 수 있다.
- 사용자 임의 사용/교환은 초기 범위에서 제공하지 않고, 사용/교환/룰렛 항목 추가는 관리자가 최종 처리한다.
- 정책 값은 코드에 고정하지 않고 관리자가 조정할 수 있게 한다.
- 방송 재미를 위한 점수이므로 금전성, 환전성, 현금 등가 표현은 금지한다.

## 2. 상태 모델

업보/쿠폰/리워드 상태:

| 상태 | 의미 |
| --- | --- |
| `OWNED` | 사용자가 보유 중 |
| `USED` | 관리자가 사용 처리함 |
| `CONVERTED` | 호감도로 전환됨 |
| `CORRECTED` | 관리자 보정 거래로 정정됨 |

상태 정책:

- 사용자는 상태를 직접 변경할 수 없다.
- `USED`, `CONVERTED`, `CORRECTED` 처리는 관리자 또는 시스템 정책에 의해서만 발생한다.
- 결과 삭제나 취소 상태는 두지 않는다.
- 잘못된 상태 변경은 새 보정 거래와 상태 변경 이력으로 정정한다.

## 3. 관리자 처리 원칙

관리자는 다음 작업을 수행할 수 있다.

- 호감도 지급 또는 차감
- 업보/쿠폰/리워드 지급
- 호감도를 차감해 쿠폰/리워드 구매 처리
- 쿠폰/리워드 구매와 동시에 사용 완료 처리
- 잔액이 부족한 사용자에게 대출성 음수 잔액 생성
- 잘못된 거래를 별도 보정 거래로 정정

정정 원칙:

- 원장 거래는 삭제하지 않는다.
- 잘못된 거래는 원본 거래와 연결된 별도 보정 거래로 정정한다.
- 예: 잘못 지급된 `호감도 +500`은 관리자가 `호감도 지급 -500` 또는 수동 조정 `-500`으로 보정한다.
- 이미 관리자가 사용 처리한 항목을 되돌려야 할 때도 자동 회수가 아니라 관리자 보정 거래로 처리한다.
- 정정 거래는 사용자 히스토리에 공개한다.
- 관리자 내부 메모만 사용자에게 공개하지 않는다.

## 4. 사용자 조회

사용자는 본인 데이터만 조회할 수 있고, 관리자는 전체 사용자 데이터를 조회할 수 있다.

검색 권한:

- 일반 사용자는 검색 기능을 사용하지 않으며, 본인 페이지로 직접 진입해 본인 데이터만 조회한다.
- 관리자는 닉네임 부분 문자열 일치(ILIKE 등)로 다른 사용자를 검색할 수 있다.
- 호감도 보드(전체 랭킹)는 인증 사용자에게 공개되며 닉네임/호감도가 노출된다.

사용자에게 보여야 하는 항목:

- 현재 호감도 (음수일 수 있으며, 음수일 때는 그대로 표시하고 빨간색 등 시각 시그널로 구분한다. 호감도 보드에서도 동일하게 처리한다.)
- 전체 호감도 변경 히스토리
- 전체 업보/쿠폰/리워드 변경 히스토리
- 누적 업보 반영 합계
- 최근 업보 내역
- 업보 라벨 또는 결과명
- 호감도 변환 점수
- 반영 전/후 호감도
- 반영 출처: `UPBO_MANUAL`, `UPBO_ROULETTE`
- 정정 여부와 원본 거래 연결
- 실행 주체. 예: 시스템, 관리자, 룰렛
- 근거 메모 중 공개 설명
- 마이페이지 마지막 조회 시점 이후 새로 발생한 보유/정정 항목의 미확인 배지

사용자 표시 예시:

| 일시 | 업보 | 호감도 반영 | 반영 후 |
| --- | --- | ---: | ---: |
| 2026-05-07 21:30 | 룰렛: 쓰담 | +10 | 132 |
| 2026-05-07 22:10 | 룰렛: 업보 | -5 | 127 |
| 2026-05-07 22:15 | 정정: 룰렛 업보 보정 | +5 | 132 |

### 4.1 미확인 배지 정책

- 사용자는 마이페이지에 마지막 진입한 시각(`lastSeenAt`)을 기준으로, 그 이후 발생한 신규 보유 항목과 정정 거래의 갯수를 미확인 배지로 표시한다.
- 페이지 진입 시 `lastSeenAt`을 갱신해 자동으로 읽음 처리한다.
- 별도 채팅 알림이나 외부 푸시 알림은 보내지 않는다.
- 같은 메커니즘으로 룰렛 결과 정정도 사용자에게 노출한다.
- 미확인 배지는 best-effort 인지 보조이며 정확한 알림 보장이 아니다. 다중 기기 동시 진입 시 한 기기에서 읽음 처리되면 다른 기기에서는 새 항목을 누락 표시할 수 있다. 엄격한 알림 보장이 필요해지면 `lastSeenAt` 단일값 대신 항목별 read marker로 확장한다.

### 4.2 보유 목록 표시

- 본인 페이지의 보유 업보/쿠폰/리워드 목록은 기본 `createdAt DESC`(최신순)으로 정렬한다.
- 상태 필터 탭으로 `OWNED`, `USED`, `CONVERTED`, `CORRECTED`를 전환할 수 있으며 기본 탭은 `OWNED`이다.

## 5. 원장 모델 후보

현재 `FavoriteHistoryEntity`는 변경 후 호감도와 사유만 저장한다. 최종 포인트 제도에서는 다음 필드가 필요하다.

- `ledgerId`: 변경 이력 ID
- `userId`: 대상 사용자
- `nickNameSnapshot`: 거래 발생 당시 표시 닉네임
- `delta`: 증감값
- `balanceAfter`: 변경 후 잔액
- `reason`: 표시용 사유
- `sourceType`: `ATTENDANCE`, `COMMAND`, `ADMIN`, `UPBO_MANUAL`, `UPBO_ROULETTE`, `SHEET_MIGRATION`, `DONATION`, `SUBSCRIPTION`, `MINI_GAME`, `REWARD_USE`
- `sourceId`: 원본 이벤트 ID
- `displayCategory`: 사용자 화면 분류. 예: `업보`, `출석`, `관리자 보상`
- `publicDescription`: 사용자에게 보여줄 설명
- `privateMemo`: 관리자 전용 메모
- `correctionOfLedgerId`: 정정 거래인 경우 원본 원장 ID
- `actorId`: 관리자 또는 시스템 실행 주체
- `idempotencyKey`: 중복 지급 방지 키
- `createdAt`: 발생 시각

## 6. 동시성과 정합성

- 호감도 잔액(`FavoriteEntity.balance`) 갱신은 사용자 단위로 비관적 락(`SELECT ... FOR UPDATE`)을 걸어 직렬화한다.
- 동시 다발적 거래(출석 적용, 룰렛 회차 반영, 관리자 수동 조정 등)가 들어와도 한 사용자 잔액은 한 번에 하나의 거래만 갱신한다.
- 잔액과 원장 합계의 정합성은 별도 정기 batch나 거래별 즉시 검증으로 검사하지 않는다. 비관적 락과 단일 트랜잭션 내 원자적 갱신을 신뢰한다.
- 이상 발견 경로:
  - 사용자 제보 (마이페이지에서 잔액·히스토리 차이 인지)
  - 관리자 수동 점검 (의심 케이스 발견 시 호감도 조회로 검증)
  - logfmt 운영 로그 검색 (`level=ERROR` 또는 `action=favorite.adjust` 등을 Grafana/Loki에서 점검)
- 발견된 불일치는 관리자가 보정 거래로 정정한다.
- 다회차 룰렛의 트랜잭션 경계는 [치지직 후원 룰렛과 OBS 오버레이](roulette-overlay.md) 6.1을 따른다.

## 7. 사용자 탈퇴와 데이터 보존

- CHZZK는 사용자 탈퇴/채널 폐쇄 이벤트를 본 시스템에 통보하지 않는다.
- 시스템은 별도 탈퇴 처리 기능을 제공하지 않는다. CHZZK 인증이 끊긴 사용자는 자연스럽게 비활성 상태가 된다.
- 호감도/업보/원장 데이터는 그대로 보존하며 닉네임 자동 마스킹도 수행하지 않는다.
- 개인정보 처리 정책 수요가 발생하면 별도 설계로 다룬다.

## 8. 결정 상태

현재 포인트 정책의 핵심 의사결정은 완료되었다. 상세 획득/사용/교환 정책은 [호감도/업보/쿠폰 카탈로그](reward-catalog.md), 후원 룰렛과 OBS 오버레이 정책은 [치지직 후원 룰렛과 OBS 오버레이](roulette-overlay.md)를 따른다.
