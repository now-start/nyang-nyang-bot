## 1. 범위

룰렛은 치지직 후원 메시지 또는 운영자 수동 실행으로 업보/호감도/쿠폰/방송 리워드를 발생시키는 기능이다. 서버가 결과를 먼저 확정하고 원장과 보유 목록에 즉시 반영하며, OBS 오버레이는 확정 결과를 애니메이션으로 표시만 한다.

## 3. 관리자 룰렛 설정

- 룰렛 테이블은 활성/비활성 상태를 가진다.
- 비활성 상태에서는 후원 명령어가 들어와도 룰렛을 실행하지 않는다.
- 관리자는 룰렛 항목을 생성/수정/비활성화할 수 있다.
- 관리자는 항목 표시 순서를 조정할 수 있다.
- 룰렛 항목 수에는 하드 제한을 두지 않는다.
- 관리자 화면에서는 항목 6-8개 구성을 권장값으로 안내하고, 항목이 많아지면 OBS/모바일 가독성 경고만 표시한다.
- 항목별 확률은 소수점 단위로 입력할 수 있어야 한다.
- 확률은 최대 소수점 넷째 자리까지 입력 가능하게 둔다.
- 전체 확률 합계가 100%가 아닌 룰렛 확률표는 활성화할 수 없다.
- `꽝` 항목은 필수 항목으로 두며 삭제하거나 0%로 설정할 수 없다.
- 룰렛 1회 금액과 명령어를 관리자가 설정할 수 있어야 한다.
- 룰렛 실행 회차는 제한하지 않는다. 동일 사용자가 여러 번 후원해도 차단 사용자 정책에만 걸린다.
- 후원자는 룰렛 금액, 항목, 확률을 수정할 수 없고 관리자 설정값에 맞춰서만 실행된다.
- 룰렛 확률표는 버전으로 관리한다. 후원 이벤트에는 실행 당시 확률표 버전을 저장한다.
- 룰렛 확률표는 전체 공개한다.

### 3.1 활성화 전 시뮬레이션

- 관리자는 룰렛 확률표를 활성화하기 전에 가상 1만 회 시뮬레이션 분포 미리보기를 확인할 수 있어야 한다.
- 미리보기는 관리자 화면에서 즉시 실행하며, 실제 원장이나 보유 목록에는 영향을 주지 않는다.
- 미리보기는 항목별 기대 확률, 시뮬레이션 결과 분포, 절대 차이를 함께 노출한다.
- 운영 중 실 결과 통계 기반의 자동 편향 알람은 두지 않는다. 사후 점검은 관리자가 결과 조회 화면에서 수동으로 수행한다.

결과 타입:

- `NONE`
- `FAVORITE_DELTA`
- `UPBO_REWARD`
- `COUPON`
- `STREAM_MISSION`
- `PARTICIPATION_PRIORITY`

## 4. 치지직 후원 룰렛 흐름

치지직 세션 API는 후원 이벤트 구독을 제공하고, 후원 이벤트 메시지에는 후원 종류, 채널 ID, 후원자 채널 ID, 후원자 닉네임, 후원 금액, 후원 메시지, 이모티콘 정보가 포함된다.

확장 흐름:

1. 앱이 치지직 세션의 후원 이벤트를 구독한다.
2. 후원 이벤트의 `donationText`에서 `!룰렛` 등 설정된 명령어를 파싱한다.
3. 후원 메시지에 룰렛 명령어가 없으면 룰렛을 실행하지 않는다.
4. 명령어가 있으면 관리자 설정 룰렛 금액, 차단 사용자, 이벤트 활성 상태를 검증한다.
5. 룰렛 회차는 `floor(후원 금액 / 관리자 설정 룰렛 금액)`으로 계산한다.
6. 회차가 1 이상이면 서버가 후원자 본인을 대상자로 실행 회차 수만큼 결과를 먼저 확정한다.
7. 확정 결과는 실행 당시 룰렛 테이블 버전, 회차 번호, 후원 이벤트 ID와 함께 저장한다.
8. 서버 확정 즉시 호감도 원장과 사용자 보유 목록에 반영한다.
9. OBS 오버레이는 오버레이 토큰으로 HTTP 이벤트를 조회해 확정 결과를 표시한다.
10. 동일 후원 이벤트가 재처리되어도 중복 반영되지 않도록 이벤트 ID 또는 idempotency key를 사용한다.

대상 정책:

- 치지직 후원 룰렛 결과는 후원자 본인에게만 적용한다.
- 후원 메시지로 다른 대상자를 지정하는 기능은 초기 범위에서 제외한다.
- 대상자 지정 기능이 필요해질 경우 관리자 설정으로 별도 활성화 여부를 둔다.

### 4.1 명령어 매칭 정책

- 후원 메시지(`donationText`)는 공백/줄바꿈 단위로 토큰을 쪼갠다.
- 토큰 중 하나라도 관리자 설정 명령어와 정확히 일치할 때만 룰렛을 실행한다.
- 예: `안녕 !룰렛 부탁드려요` → 매칭. `!룰렛!`, `!룰렛이다` 같은 인접 결합 토큰은 매칭하지 않는다.
- 부분 문자열 포함 매칭이나 정규식 매칭은 사용하지 않는다.
- 관리자 화면에서는 명령어 입력 시 위 규칙을 안내한다.

### 4.2 후원 시점 스냅샷

- 후원 이벤트 수신 즉시 활성 룰렛 테이블의 `version`, 활성 상태, 항목 라벨/확률/결과 타입을 RouletteEvent에 스냅샷으로 저장한다.
- 룰렛 결과 확정과 원장 반영은 모두 이 스냅샷을 기준으로 한다.
- 후원 처리 중에 관리자가 룰렛 테이블을 비활성화하거나 항목/확률을 변경해도 진행 중인 후원은 스냅샷대로 끝까지 실행한다.
- 후원 도착 시점에 이미 비활성화된 테이블은 스냅샷을 만들지 않고 룰렛을 실행하지 않는다.

### 4.3 중복 방지 키

- idempotency key는 CHZZK가 부여하는 후원 이벤트 ID 단일 값을 사용한다.
- 동일 후원 이벤트 ID로 재수신되면 모두 무시하고 원장/보유 목록에 반영하지 않는다.
- 메시지 수정·세션 재연결로 인한 재수신도 같은 정책을 따른다.

### 4.4 환불·취소 이벤트

- CHZZK는 후원 환불/취소 이벤트를 별도로 발행하지 않는 것으로 간주한다.
- 시스템은 환불 이벤트를 자동 처리하지 않는다.
- 사후 환불 사실이 알려지면 관리자가 일반 보정 거래로 정정한다.

### 4.5 자동 안내 채팅

- 룰렛 결과 자체뿐 아니라 `결과는 마이페이지에서 확인하세요` 류의 안내 채팅도 시스템이 자동으로 발송하지 않는다.
- 후원자는 OBS 오버레이, 본인 마이페이지, `!룰렛결과` 채팅 명령어로 결과를 확인한다.

## 5. 금액/회차 정책

- 룰렛 명령어는 관리자 설정값으로 관리한다. 기본값은 `!룰렛`이다.
- 룰렛 1회 금액은 관리자가 설정한다.
- 후원 메시지에 룰렛 명령어가 없으면 후원 금액과 관계없이 룰렛을 실행하지 않는다.
- 후원 메시지에 룰렛 명령어가 있으면 `후원 금액 / 룰렛 1회 금액`의 정수 몫만큼 룰렛을 실행한다.
- 예: 룰렛 1회 금액이 1000원이고 후원 금액이 11100원이며 메시지가 비어 있으면 룰렛은 실행되지 않는다.
- 예: 룰렛 1회 금액이 1000원이고 후원 금액이 11100원이며 메시지에 `!룰렛`이 있으면 11회 실행한다.
- 나머지 금액은 룰렛 회차로 이월하지 않는다.
- 후원 1건당 최대 실행 회차는 별도 상한을 두지 않는다. CHZZK 후원이 사실상 환불 불가 채널이라는 점을 전제로, 운영상 폭주 방지는 차단 사용자 정책과 관리자 모니터링으로 처리한다.
- 회차 수가 운영 임계치(기본 100회)를 넘는 후원이 들어오면 logfmt AUDIT 로그에 `action=roulette.high_round` 형태로 표기해 사후 모니터링이 가능하게 한다. 자동 차단·지연 처리는 두지 않는다. 임계치는 관리자 설정값으로 둔다.

## 6. 결과 반영/정정 정책

- 서버는 전체 결과를 먼저 확정한다.
- 서버 확정 즉시 호감도 원장과 사용자 보유 목록에 반영한다.
- 오버레이 표시 완료 여부는 원장 반영의 선행 조건이 아니다.
- 룰렛 결과는 채팅에 자동 공지하지 않고, 화면 오버레이와 사용자 히스토리에서 확인한다.
- 룰렛 결과 자체는 삭제하거나 취소 상태로 되돌리지 않는다.
- 잘못된 룰렛 결과, 테스트 후원, 오작동 등은 관리자가 별도 보정 거래를 생성해 정정한다.
- 예: `호감도 +500`이 잘못 지급된 경우 관리자는 `호감도 지급 -500` 또는 수동 조정 `-500`으로 보정한다.
- 쿠폰/업보/미션 결과의 사용 여부도 사용자가 임의로 확정하지 않는다. 관리자가 최종 사용 처리 또는 보정 처리한다.
- 이미 관리자가 사용 처리한 항목을 되돌려야 할 때도 자동 회수가 아니라 관리자 보정 거래로 처리한다.
- 정정 거래는 원본 룰렛 이벤트와 연결하고 사용자 히스토리에 공개한다.
- 정정 거래가 생성되면 사용자 마이페이지의 미확인 배지 카운트를 증가시켜 사용자가 인지할 수 있게 한다.

### 6.1 다회차 트랜잭션 경계

- N회차(예: 11회) 룰렛은 두 단계로 나눠 처리한다.
  1. **결과 일괄 확정**: 전체 N회 결과를 하나의 트랜잭션에서 `RouletteRoundResultEntity`로 `CONFIRMED` 상태 일괄 저장한다.
  2. **회차별 반영**: 각 회차의 호감도 원장 반영과 보유 목록 추가는 회차별 별도 트랜잭션으로 분리해 `APPLIED` 상태로 전환한다.
- 분리 이유: 결과 확정의 원자성은 보장하되, 다회차 반영 도중 일부 회차가 실패해도 재처리 단위를 작게 가져가기 위해서이다.
- `CONFIRMED`까지만 도달한 회차는 잡(scheduler) 또는 관리자 도구로 재반영을 시도한다.
- 모든 호감도 잔액 갱신은 사용자별 비관적 락(`SELECT ... FOR UPDATE`)으로 직렬화한다.

## 7. OBS 오버레이 정책

오버레이 기본 동작:

- 관리자는 관리자 페이지에서 OBS 오버레이 토큰을 발급한다.
- OBS 브라우저 소스는 `/overlay/roulette#token=...` 형태의 URL을 호출해 오버레이 페이지를 연다.
- URL fragment의 토큰은 초기 HTTP 요청에 포함되지 않으므로 서버 접근 로그에 남지 않는다.
- 오버레이 페이지는 투명 배경의 HTML/JavaScript 화면이며, 표시할 룰렛 이벤트가 없을 때는 아무 UI도 보여주지 않는다.
- 오버레이 JavaScript는 fragment에서 읽은 토큰을 `Authorization: Bearer` 헤더에 넣어 HTTP 기반 이벤트 조회를 수행한다.
- 구현 방식은 헤더 인증이 가능한 `fetch` 기반 long polling 또는 streaming fetch를 우선 사용한다.
- 오버레이는 결과를 결정하지 않고 표시만 담당한다.

애니메이션:

- JavaScript 애니메이션은 무작위로 돌아가는 느낌을 주되, 각 회차의 최종 정지 위치는 서버가 전달한 확정 결과와 반드시 일치해야 한다.
- 순차 애니메이션으로 표시할 최대 회차 수 `N`은 관리자가 설정한다.
- 예: `N=5`이고 룰렛이 11회 실행되면, 5회는 순차 애니메이션으로 보여주고 6회는 요약 결과로 표시한다.
- 서버는 전체 11회 결과를 모두 먼저 확정하고, 오버레이는 표시 방식만 나눠 처리한다.

이벤트 유효시간과 재송출:

- 오버레이 이벤트의 기본 표시 유효시간은 120초이다.
- 오버레이가 120초 안에 이벤트를 가져가면 애니메이션을 표시한다.
- 120초 안에 오버레이가 이벤트를 가져가지 못하면 해당 오버레이 이벤트는 `MISSED` 상태로 남기고 자동 재생하지 않는다.
- 관리자는 `MISSED` 또는 이미 표시된 룰렛 이벤트를 오버레이에 재송출할 수 있다.
- 재송출은 화면 애니메이션만 다시 보내며, 호감도 원장과 사용자 보유 목록은 다시 반영하지 않는다.
- 오버레이는 애니메이션 표시 완료 후 서버에 표시 완료 상태를 HTTP로 보고한다.
- 오버레이 연결이 끊기거나 OBS가 꺼져도 서버 확정 결과와 이미 반영된 원장/보유 목록은 취소하지 않는다.

오버레이 토큰:

- 오버레이 토큰은 쉽게 폐기/재발급할 수 있어야 한다.
- 재발급 시 기존 토큰은 즉시 무효화하고, 새 OBS URL을 다시 복사할 수 있어야 한다.
- 토큰 원문은 발급 또는 재발급 직후 한 번만 보여주며, DB에는 해시만 저장하고 로그에는 남기지 않는다.

## 8. 투네이션 룰렛 벤치마크 반영

- 투네이션은 크리에이터가 룰렛 금액, 상품, 확률을 지정하고 도네이터가 이를 임의로 수정하지 못하는 구조를 사용한다.
- 투네이션은 룰렛 상품 확률 합계 100%, `꽝` 필수, 상품 최대 20개, 개인/전체 후원 횟수 제한을 제공한다.
- 냥냥봇 MVP에서는 스트리머 운영 자유도를 우선해 항목 수 제한과 개인/전체 후원 횟수 제한을 채택하지 않는다.
- 냥냥봇은 결제 플랫폼 자체 룰렛이 아니라 치지직 후원 메시지 기반 룰렛이므로, 위 설정 모델만 참고하고 실행 트리거는 `!룰렛` 같은 명령어 파싱으로 처리한다.
