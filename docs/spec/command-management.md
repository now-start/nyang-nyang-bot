# 관리자 명령어 관리 PRD

- 문서 상태: Draft v1
- 작성일: 2026-07-03
- 상위 문서: [Nyang-Nyang Bot Spec](index.md)
- 관련 요구사항: FR-022
- 기준 코드베이스: `CommandType`, `ChatService`, `Command`

## 1. 배경

현재 채팅 명령어는 Java enum `CommandType`과 Spring Bean `Command` 구현체에 고정되어 있다. `!호감도`, `!룰렛결과` 같은 시스템 명령은 코드 배포 없이는 추가, 비활성화, 쿨타임 변경이 어렵다.

방송 운영에서는 안내 문구, 이벤트 설명, 링크, 규칙 같은 반복 응답이 자주 바뀐다. 관리자가 매번 배포 없이 안전하게 명령어를 추가하고 조정할 수 있어야 한다.

## 2. 문제 정의

- 간단한 안내 명령어도 코드 수정과 배포가 필요하다.
- 명령어별 쿨타임 정책이 문서에는 있지만 운영 화면에서 변경할 수 없다.
- 기존 시스템 명령어와 후원 룰렛 명령어가 같은 키워드를 쓰는 충돌을 사전에 막기 어렵다.
- 어떤 명령어가 활성 상태인지, 누가 마지막으로 바꿨는지 운영자가 한 화면에서 보기 어렵다.
- 임의 실행형 명령어를 허용하면 권한, 보안, 원장 정합성 문제가 커진다.

## 3. 목표

1. 관리자가 웹 관리자 화면에서 텍스트 응답형 채팅 명령어를 생성, 수정, 비활성화할 수 있게 한다.
2. 기존 시스템 명령어는 코드 동작을 유지하되, 활성 여부와 쿨타임을 관리할 수 있게 한다.
3. 명령어 생성 시 기존 명령어와 룰렛 후원 트리거가 충돌하지 않게 검증한다.
4. 메시지 템플릿은 허용된 읽기 전용 변수만 치환한다.
5. 명령어 실행은 사용자별 쿨타임과 권한 정책을 따른다.
6. 관리자 변경 이력은 최소한 운영 로그와 수정자/수정시각으로 추적 가능해야 한다.

## 4. 비목표

- 임의 JavaScript, SQL, shell 실행
- 관리자가 임의 URL을 입력하는 webhook 또는 외부 HTTP 호출
- 채팅 메시지로 명령어를 등록/삭제하는 기능
- 일반 사용자 또는 매니저가 관리자 화면 없이 명령어를 수정하는 기능
- 관리자 작성 템플릿으로 호감도 원장, 업보, 룰렛 결과를 직접 변경하는 기능
- 단, allow-list에 등록된 `TRIGGER` action은 기존 도메인 use case를 실행할 수 있다.
- 음악 신청, 게임 전적 조회, 미니게임 같은 도메인별 고급 명령어
- 초기 단계의 복수 채널 멀티테넌시

## 5. 사용자와 권한

| 사용자 | 권한 |
| --- | --- |
| 일반 사용자 | 활성 명령어 호출 |
| 관리자 | 명령어 생성, 수정, 비활성화, 미리보기 |
| 시스템 | CHZZK 채팅 수신, 명령어 매칭, 쿨타임 판정, 응답 전송 |

초기 권한은 기존 정책과 동일하게 `AuthorizationAccount.admin = true` 단일 관리자 기준을 사용한다. `MANAGER`, `STREAMER`, `VIEWER` 같은 세부 권한은 명령어 필드에 확장 여지를 두되 MVP에서는 `USER`, `ADMIN`만 실제 적용한다.

## 6. 명령어 유형

| 유형 | 설명 | 구현 범위 |
| --- | --- | --- |
| `TEXT` | 관리자가 작성한 고정 문구와 변수 치환 응답 | P1 |
| `TRIGGER` | 코드로 구현된 내부 동작을 실행하는 트리거. 예: `!호감도`, `!룰렛결과`, 후원 메시지 기반 룰렛 | P1 |
| `TIMER` | 일정 조건마다 자동 발송되는 메시지 | P2 |

유형은 특정 기능명이 아니라 실행 전략 기준으로 둔다. `!호감도`, `!룰렛결과`, 룰렛 후원 명령어는 모두 별도 전용 타입을 만들지 않고 `TRIGGER`로 다룬다.

세 유형은 모두 PRD와 데이터 모델 범위에 포함한다. 단, P1 구현은 `TEXT`, `TRIGGER`를 우선 완료하고, `TIMER`는 같은 모델 위에서 P2로 확장한다.

`TRIGGER` 명령어의 실제 동작은 코드가 담당한다. 관리자는 응답 로직을 직접 편집하지 않고, 트리거와 운영 정책만 조정한다.

`TEXT`, `TRIGGER`는 채팅 입력의 첫 토큰이 트리거와 정확히 일치하면 실행된다.

`TIMER`는 채팅 입력으로 실행되지 않고, 분 단위 주기와 조건을 만족할 때 시스템이 발송한다.

## 7. 핵심 사용자 흐름

### 7.1 텍스트 명령어 생성

1. 관리자가 `/admin/commands`에 진입한다.
2. `명령어 추가`를 선택한다.
3. 트리거, 메시지 템플릿, 쿨타임을 입력한다.
4. 시스템이 형식, 중복, 룰렛 후원 트리거 충돌, 변수 유효성을 검증한다.
5. 관리자가 미리보기로 샘플 응답을 확인한다.
6. 저장 후 활성화하면 다음 채팅부터 명령어가 동작한다.

### 7.2 시스템 명령어 운영 정책 변경

1. 관리자가 기존 `!호감도` 또는 `!룰렛결과` 행을 선택한다.
2. 활성 여부와 쿨타임을 조정한다.
3. 시스템은 시스템 명령어 키와 실제 action 연결이 끊기지 않는지 검증한다.
4. 저장 후 설정된 트리거가 같은 시스템 action을 실행한다.

### 7.3 명령어 실행

1. 사용자가 CHZZK 채팅에 `!규칙` 같은 명령어를 입력한다.
2. 시스템은 첫 토큰만 명령어로 해석한다.
3. 활성 명령어와 정확히 일치하지 않으면 일반 채팅으로 처리한다.
4. 권한 또는 쿨타임에 걸리면 응답하지 않는다.
5. `TEXT` 명령어는 메시지 템플릿을 치환해 채팅으로 응답한다.
6. `TRIGGER` 명령어는 연결된 내부 action을 실행한다.

## 9. 명령어 정책

### 9.1 형식

- `TEXT`, `TRIGGER`의 트리거는 `!`로 시작한다.
- `TEXT`, `TRIGGER`의 트리거는 공백을 포함할 수 없다.
- `TEXT`, `TRIGGER`의 트리거 길이는 `!` 포함 2자 이상 20자 이하이다.
- `TEXT`, `TRIGGER`의 트리거에는 제어문자, 줄바꿈, 탭을 사용할 수 없다.
- `CHAT_MESSAGE`는 첫 토큰 정확 일치만 명령어로 본다.
- `DONATION_MESSAGE`의 `TRIGGER/action_key=ROULETTE_DONATION`은 공백 기준 토큰 중 하나라도 정확히 일치하면 실행 대상으로 본다.
- 비활성 명령어는 매칭 대상에서 제외한다.
- 별칭은 MVP에서 지원하지 않는다.

### 9.2 충돌

모든 명령어 유형은 하나의 트리거 namespace를 공유한다. 저장 시 다음 중복을 차단한다.

- 기존 `TEXT` 트리거와 중복
- 기존 `TRIGGER` 트리거와 중복

검증 적용 위치:

- 명령어 생성/수정/활성화
- 룰렛 테이블 생성/수정/활성화
- 기존 룰렛 테이블의 명령어 변경

구현 요구:

- 사전 검증과 저장은 같은 트랜잭션에서 수행한다.
- 트리거는 저장 전에 앞뒤 공백 제거, Unicode NFC, `Locale.ROOT` lower-case로 정규화해 `trigger`에 저장한다.
- `trigger`는 활성/비활성 상태와 관계없이 전역에서 중복될 수 없다.
- 같은 트리거를 다시 쓰려면 새 row를 만들지 않고 기존 row를 수정하거나 재활성화한다.
- 최종 중복 방지는 `trigger` unique index가 담당한다. `TIMER`의 `trigger=null`은 여러 row를 허용한다.
- 명령어 트리거와 룰렛 후원 트리거는 `chat_command` 단일 테이블에 기록한다.
- `chat_command.trigger`가 명령어 매칭의 authoritative source이다.
- 기존 `roulette_table.command`는 전환 단계에서만 backfill 원천 또는 표시용 사본으로 사용하고, 런타임 매칭은 `chat_command`를 읽는다.
- 룰렛 명령어 변경은 `chat_command`와 룰렛 테이블 저장을 같은 트랜잭션으로 묶어 두 값이 갈라지지 않게 한다.

### 9.3 쿨타임

- 기본 사용자별 쿨타임은 30초이다.
- 쿨타임은 채팅 메시지로 실행되는 `TEXT`와 `TRIGGER`에만 적용한다.
- `action_key=ROULETTE_DONATION`처럼 후원 메시지로 실행되는 `TRIGGER`에는 명령어 쿨타임을 적용하지 않는다.
- 후원 룰렛은 후원 이벤트 idempotency, 차단 사용자 정책, 룰렛 활성 상태로만 실행 여부를 결정하며 명령어 쿨타임 때문에 무시하지 않는다.
- MVP에서는 글로벌 채널 쿨타임을 두지 않는다.
- 쿨타임은 5초 이상 3,600초 이하로만 저장할 수 있다. 0초는 허용하지 않는다.
- 쿨타임 차단은 채팅 응답을 보내지 않는 방식으로 처리한다.

## 10. 메시지 템플릿 변수

기본 제공 변수:

| 변수 | 의미 | 데이터 출처 | 사용 유형 |
| --- | --- | --- | --- |
| `{nickname}` | 호출자 닉네임 | CHZZK 채팅 payload | `TEXT` |
| `{command}` | 호출된 명령어 토큰 | 채팅 메시지 | `TEXT` |
| `{args}` | 명령어 뒤 원문 인자 | 채팅 메시지 | `TEXT` |
| `{arg1}` | 첫 번째 인자 | 채팅 메시지 | `TEXT` |
| `{arg2}` | 두 번째 인자 | 채팅 메시지 | `TEXT` |
| `{favorite}` | 호출자 현재 호감도 | `favorite_account.favorite` | `TEXT` |
| `{date}` | 서버 기준 날짜 | Clock | `TEXT`, `TIMER` |
| `{time}` | 서버 기준 시간 | Clock | `TEXT`, `TIMER` |
| `{datetime}` | 서버 기준 날짜와 시간 | Clock | `TEXT`, `TIMER` |

규칙:

- 저장 시 알 수 없는 변수는 오류로 처리한다.
- 변수 조회 실패 시 명령어 전체를 실패시키지 않고 안전한 대체 문구를 사용한다.
- 호출자 기반 변수인 `{nickname}`, `{command}`, `{args}`, `{arg1}`, `{arg2}`, `{favorite}`는 `TIMER`에서 사용할 수 없다. 저장 단계에서 오류로 처리한다.
- `{arg1}`, `{arg2}`는 공백 기준으로 파싱하며, 없는 값은 빈 문자열로 치환한다.
- 채팅 응답 `TEXT`에서 `{args}`, `{arg1}`, `{arg2}`를 사용할 때는 사용자 입력 echo sanitizer를 반드시 거친다.
- 사용자 입력 echo sanitizer는 제어문자 제거, 공백 접기, 변수별 최대 60자 제한, URL-like 토큰 비활성화, 멘션 토큰 비활성화를 수행한다.
- DB 조회 변수는 `{favorite}`만 제공한다. 업보/쿠폰, 룰렛 결과, 랭킹 변수는 P1에서 제공하지 않는다.
- 메시지 템플릿은 상태 변경 use case를 호출하지 않는다.
- 변수 조회는 채팅 payload, Clock, 호출자 `favorite_account.favorite`만 사용한다.
- 변수별 resolver는 독립적으로 실패를 격리하고, 실패한 변수는 `-`로 치환한다.
- 관리자 미리보기는 샘플 사용자와 실제 변수 치환 규칙을 공유한다.
- 저장 메시지 템플릿과 렌더링 결과는 각각 최대 300자까지 허용한다.
- 저장, 미리보기, 실제 실행은 같은 렌더러와 같은 길이 검증을 사용한다.

## 11. 관리자 UI 요구사항

필수 화면:

- 명령어 목록: 검색, 유형 필터, 활성 상태 필터, 쿨타임, 수정시각
- 명령어 생성/수정 폼: 트리거, 유형, 메시지 템플릿, 쿨타임
- 미리보기: 샘플 닉네임, 샘플 인자, 현재 시간 기준 응답 확인
- 충돌 표시: 어떤 명령어 또는 룰렛 설정과 충돌하는지 inline으로 표시
- 비활성화 확인: 삭제 대신 비활성화를 기본 액션으로 제공

UI 원칙:

- 관리자 반복 작업 화면이므로 테이블 중심의 조밀한 화면으로 구성한다.
- 명령어 본문이 길면 2줄 clamp 또는 상세 펼침을 사용한다.
- 위험 동작은 삭제가 아니라 비활성화로 표현한다.
- 모바일 360px에서도 트리거 토큰과 활성 토글이 겹치지 않아야 한다.

`chat_command`

| 필드 | 설명 |
| --- | --- |
| `id` | 명령어 ID |
| `type` | `TEXT`, `TRIGGER`, `TIMER` |
| `trigger` | 채팅 또는 이벤트 메시지에서 매칭할 단일 토큰. `TIMER`는 null 가능 |
| `action_key` | `TRIGGER`가 실행할 내부 action 식별자. 예: `FAVORITE_STATUS`, `ROULETTE_RESULT`, `ROULETTE_DONATION` |
| `message_template` | `TEXT` 또는 `TIMER`가 발송할 메시지 템플릿 |
| `timer_interval_minutes` | `TIMER` 반복 주기. 기본 단위는 분 |
| `timer_min_chat_count` | 현재 타이머 interval 동안 필요한 최소 채팅 수 |
| `active` | 활성 여부 |
| `required_role` | MVP는 `USER`, `ADMIN` |
| `user_cooldown_seconds` | 사용자별 쿨타임 |
| `created_by`, `updated_by` | 관리자 channelId 또는 system |
| `create_date`, `modify_date` | BaseEntity 기준 |

제약:

- `trigger`가 있는 row는 활성 여부와 관계없이 전체에서 유일해야 한다.
- `TIMER`는 트리거 없이 동작할 수 있으므로 `trigger=null`을 허용한다.
- `TEXT`, `TRIGGER`는 하나의 트리거를 가진다. 별칭은 지원하지 않는다.
- `TIMER`는 트리거 없이 동작할 수 있다.
- `TRIGGER` 명령어는 `action_key`가 필요하다.
- `action_key`는 내부 allow-list 값만 저장할 수 있고 생성 후 변경할 수 없다.
- `ROULETTE_DONATION`은 별도 설정값 없이 현재 활성 룰렛 테이블 1개를 사용한다.
- `TEXT` 명령어는 `message_template`이 필요하다.
- `TIMER` 명령어는 `message_template`, `timer_interval_minutes`가 필요하다.
- `timer_interval_minutes`는 5 이상 1,440 이하의 정수이며 UI와 API 모두 분 단위로 받는다.
- `timer_min_chat_count`는 1 이상 10,000 이하의 정수만 허용한다.
- `timer_min_chat_count`는 현재 런타임 interval 동안 수신한 `CHAT_MESSAGE` 이벤트 수를 기준으로 계산한다.
- 매니저, 스트리머, 일반 사용자를 구분하지 않는다. 카운트 기준은 사용자 역할이 아니라 수신 이벤트 타입이다.
- `DONATION_MESSAGE`, 시스템 내부 이벤트, 타이머 발송 결과는 타이머 채팅 수에 포함하지 않는다.
- 같은 채팅 소켓으로 봇이 보낸 메시지가 다시 들어오는 경우에는 설정된 bot `channelId`와 일치하는 `CHAT_MESSAGE`만 제외한다.
- `TIMER` 실행 상태는 `chat_command` 정의 컬럼에 저장하지 않는다. P2 구현 시 런타임 상태로 interval 시작 시각과 채팅 수를 관리한다.
- `TIMER`가 활성화되거나 타이머 설정이 바뀌면 런타임 상태에서 새 interval을 시작한다.
- 타이머 interval이 끝나면 발송 여부와 관계없이 런타임 상태에서 다음 interval을 시작한다.
- 룰렛 테이블 명령어도 `TRIGGER` row로 같은 테이블에 기록한다.

## 13. API 후보

| Method | Path | 권한 | 설명 |
| --- | --- | --- | --- |
| GET | `/admin/commands` | 관리자 | 명령어 목록 조회 |
| POST | `/admin/commands` | 관리자 | 명령어 생성 |
| PATCH | `/admin/commands/{id}` | 관리자 | 명령어 수정, 활성/비활성 변경 |
| POST | `/admin/commands/preview` | 관리자 | 템플릿 미리보기 |
| POST | `/admin/commands/validate` | 관리자 | 저장 전 충돌/변수 검증 |

보안 요구:

- `POST`, `PATCH` 관리자 명령어 API는 세션 쿠키만으로 상태를 변경할 수 없게 한다.
- 구현 시 CSRF 토큰을 활성화하거나, 동일 출처 JSON 요청 검증과 별도 관리자 API 토큰 중 하나를 적용한다.
- 동일 출처 JSON 요청 검증 방식을 선택한 경우에만 `Origin`/`Referer`가 관리자 화면 origin과 다른 상태 변경 요청을 거부한다.
- 미리보기와 검증 API도 관리자 권한과 같은 보안 정책을 적용한다.

구현 착수 시 [API 명세](api.md)에 요청/응답 DTO와 에러 코드를 반영한다.

## 14. 구현 단계 제안

### Step 1. 기존 명령어 seed

- `!호감도`, `!룰렛결과`를 `TRIGGER` 명령어로 seed하고 각각 `action_key=FAVORITE_STATUS`, `ROULETTE_RESULT`로 연결한다.
- 룰렛 테이블의 `!룰렛`은 `TRIGGER` 명령어로 등록한다.
- 기존 `roulette_table.command`가 남아 있다면 `chat_command`로 backfill한 뒤 런타임 매칭 경로에서는 제거하거나 비권위 표시 필드로만 유지한다.
- 기존 `CommandType` enum은 DB 레지스트리 전환 전까지 호환 경로로 유지한다.
- `action_key`는 기존 `Command` 구현 또는 도메인 use case를 가리키는 내부 allow-list key와 맞춘다.

### Step 2. 명령어 레지스트리 전환

- `ChatService`가 enum 대신 활성 명령어 레지스트리에서 첫 토큰을 해석한다.
- 내부 action map은 `TRIGGER` 명령어 실행에만 사용한다.
- 쿨타임 key는 명령어 문자열이 아니라 명령어 ID를 기준으로 한다.

### Step 3. 텍스트 명령어 실행

- `TEXT` 명령어 renderer를 추가한다.
- 메시지 템플릿 변수 치환은 허용 목록 기반으로 처리한다.
- 변수 조회 실패는 로그로 남기고 사용자에게 안전한 대체 응답을 보낸다.

### Step 4. 타이머 실행

- `TIMER`는 분 단위 설정 주기가 지나고, 현재 interval의 채팅 수가 `timer_min_chat_count` 이상일 때만 발송한다.
- 타이머 발송 평가는 `CHAT_MESSAGE` 수신 처리 흐름에서 수행한다. 별도 scheduler가 채팅 입력 없이 주기만으로 발송하지 않는다.
- 타이머 실행 상태는 P2 구현 세부사항이며 `chat_command` 정의 테이블에는 저장하지 않는다.
- 외부 URL, SQL, shell, 임의 webhook 입력은 저장할 수 없다.

### Step 5. 관리자 UI/API

- 명령어 목록, 생성, 수정, 비활성화, 미리보기를 구현한다.
- 충돌 검증은 저장 API와 미리보기/검증 API에서 같은 정책을 사용한다.

## 15. 검증 기준

- `TEXT` 명령어 생성 후 채팅 첫 토큰이 정확히 일치할 때만 응답한다.
- `DONATION_MESSAGE` 룰렛 트리거는 메시지의 공백 기준 토큰 어디에 있어도 정확히 일치하면 실행 대상으로 처리한다.
- 비활성 명령어는 응답하지 않는다.
- 명령어는 별칭 없이 하나의 트리거만 가진다.
- 기존 `!호감도`, `!룰렛결과`는 seed 후에도 기존 내부 action을 실행한다.
- `TIMER`는 트리거 없이 분 단위 주기와 interval 채팅 수 조건으로 자동 발송된다.
- `TIMER`는 현재 interval의 `CHAT_MESSAGE` 수가 `timer_min_chat_count` 미만이면 발송하지 않는다.
- `timer_min_chat_count=0`은 저장할 수 없다.
- 명령어 저장은 기존 `TRIGGER`와 충돌할 수 없다.
- 도메인 기능 명령어 저장/활성화는 기존 `TEXT`, `TRIGGER`와 충돌할 수 없다.
- 후원 룰렛은 명령어 쿨타임 때문에 무시되지 않는다.
- 동시에 발생한 관리자 저장 요청도 DB-backed uniqueness로 하나만 성공한다.
- 일반 사용자는 관리자 명령어 API를 호출할 수 없다.
- 알 수 없는 템플릿 변수는 저장할 수 없다.
- `./gradlew test`가 통과한다.

## 16. 정책 결정

| 항목 | 결정 |
| --- | --- |
| 생성 기본값 | 신규 명령어는 `active=false`, `required_role=USER`, `user_cooldown_seconds=30`으로 만든다. `TIMER` 기본값은 `timer_interval_minutes=10`, `timer_min_chat_count=10`이다. |
| 트리거 형식 | `TEXT`, `TRIGGER`의 트리거는 앞뒤 공백 제거 후 저장하며 `!`로 시작해야 한다. 공백/제어문자는 허용하지 않고, 길이는 2자 이상 20자 이하로 제한한다. `TIMER`는 트리거 없이 동작한다. |
| 쿨타임 0초 | 허용하지 않는다. `user_cooldown_seconds`는 5 이상 3,600 이하만 저장한다. |
| 관리자 세부 권한 | MVP는 기존 `ADMIN` 단일 권한만 사용한다. |
| 명령어 삭제 | 물리 삭제는 제공하지 않는다. 비활성화만 제공한다. |
| 별칭 | 지원하지 않는다. `TEXT`, `TRIGGER`는 하나의 트리거만 가진다. |
| 중복 정책 | 비활성 명령어까지 포함해 같은 트리거는 하나만 저장한다. 같은 트리거가 필요하면 기존 row를 수정하거나 재활성화한다. |
| 메시지 길이 | 저장 메시지 템플릿과 렌더링 결과 모두 300자 이하로 고정한다. |
| 타이머 주기 | `timer_interval_minutes`는 5 이상 1,440 이하의 분 단위 정수만 허용한다. |
| 타이머 채팅 수 | `timer_min_chat_count`는 현재 interval 안의 `CHAT_MESSAGE` 수만 집계한다. 사용자 역할을 구분하지 않으며, 1 이상 10,000 이하만 허용한다. 0은 저장할 수 없다. |
| 유형 변경 | 생성 후 `type`은 변경할 수 없다. 다른 유형이 필요하면 기존 명령어를 비활성화하고 새로 만든다. |
| `TRIGGER` action 변경 | 생성 후 `action_key`는 변경할 수 없다. 다른 내부 동작이 필요하면 기존 명령어를 비활성화하고 새로 만든다. |
| 공개 목록 | 공개 명령어 목록 API와 공개 페이지 노출은 제공하지 않는다. |
